# 第二阶段工程化总结与反思

## 已完成的工程化增强

1. LLM 接入增强  
   新增统一 `llm_service.py`，支持火山引擎方舟 OpenAI 兼容接口，并保留规则模板 fallback。未配置 API Key、请求失败或模型异常时，Demo 仍可稳定运行。

2. 适岗评分解释增强  
   适岗分从单一分数升级为可解释评分：展示原始分、权重、贡献分、状态和维度解释。这样 HR 和导师可以理解“为什么是这个等级”。

3. 页面体验优化  
   首页增加推荐演示路径，页面增加生成来源提示，降低评审或演示时的理解成本。

4. 内容导出能力  
   支持 HR 看板摘要 Markdown、实习生明细 CSV、个人画像 Markdown、任务记录 CSV、周报 Markdown、风险名单 CSV 导出。

5. 质量检查入口  
   新增 `scripts/quality_check.py`，统一执行规则校验、Dashboard 数据校验、AI 功能校验、LLM fallback 校验、评分解释校验、导出校验和 Python 语法检查。

## 当前仍存在的限制

1. 数据仍为 CSV 模拟数据  
   已在第三阶段启动 SQLite 持久化：CSV 作为种子数据，SQLite 用于运行时读取和导师反馈保存。若要正式生产化，仍建议迁移 PostgreSQL。

2. 没有真实登录与权限系统  
   当前通过页面入口模拟 HR、导师、实习生视角，没有实现角色鉴权。

3. LLM 输出只增强部分内容  
   评分、风险等级仍由规则计算，这是稳定性的优点，但也意味着模型不会参与复杂人才判断。

4. 没有持久化用户输入  
   已在第三阶段补齐导师反馈保存，当前会写入 SQLite，并保留反馈历史。限制是 Streamlit Cloud 本地 SQLite 不适合长期多用户生产存储。

5. 缺少真实业务数据校准  
   评分权重和风险关键词基于业务假设，尚未经过历史实习表现数据验证。

## 是否还有值得提升的功能

有，但不建议在当前作业交付前继续扩张范围。值得规划到后续版本：

1. 数据持久化  
   使用 SQLite 或 PostgreSQL 保存导师反馈、任务进度和周报记录。

2. 角色权限  
   增加 HR、导师、实习生、招聘团队的登录与可见范围控制。

3. 历史趋势  
   增加周度趋势图，观察个人成长曲线和岗位整体变化。

4. 真实 LLM 结构化输出  
   将导师反馈分析升级为 JSON Schema 输出，并加入解析失败重试。

5. 评分模型校准  
   使用真实历史数据回看评分权重，评估适岗分与转正结果的相关性。

6. 更正式的测试体系  
   已在第三阶段启动：新增 pytest 测试和 GitHub Actions。后续可继续扩展覆盖率。

## 第二阶段收口判断

第二阶段已经完成“工程化增强”的核心目标：真实 LLM 可接入、fallback 稳定、评分解释可理解、内容可导出、质量检查可一键运行。当前项目适合进入最终演示、提交和后续小范围优化，不建议继续在本轮增加复杂功能。
